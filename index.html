核心主張

「我已經幾乎不用 Markdown 了 —— 改用 HTML。」
— Thariq Shihipar，Anthropic Claude Code 團隊
  • 超過約 100 行的 Markdown，他坦言「幾乎讀不下去」。
  • 不只個人偏好 —— Anthropic 內部已把 HTML 當成 plan、code review、設計系統、報告的預設格式。
  • 想法很簡單：讓 Claude 直接產出能排版、視覺化、可互動的東西。

他提出的五大理由

01
資訊密度
表格、CSS、SVG、程式碼、圖片一檔搞定
02
長文可讀性
Markdown 過 100 行難讀；HTML 有結構排版
03
易於分享
一個連結就能看，不必當附件寄
04
雙向互動
滑桿、旋鈕、可編輯、複製鈕
05
愉悅感
讀精緻 HTML ≠ 盯著一堆符號

代價：token，與「給機器讀」的場景

  • Token 成本：同內容 HTML 約是 Markdown 的 2–3 倍（3000 字：~2,800 → ~8,000 tokens）。
  • 機器理解 / RAG：要模型「讀進去」時 Markdown 較好（表格抽取 60.7% vs 53.6%；RAG 準確率最多 +35%）。
  • 社群出現反諷文〈The Unreasonable Ineffectiveness of HTML〉等反駁。
  • 作者本人承認 token 代價，但認為「表達力 + 讀者真的會讀」值得 —— Opus 有 100 萬 token context。

共識：不是二選一，看用途分工

／／ Markdown 勝出
當作模型「輸入」、通用指令
  • 省 token、成本低
  • 可 diff、版本好管理
  • 到處都能正常顯示
  • RAG / 結構化抽取較準
VS
</> HTML 勝出
給「人」看的最終產出
  • 報告、簡報、富結構文件
  • 版面、色彩、視覺化
  • 互動元件與分享連結
  • 長文的可讀性

重點是「配合產物的生命週期」選格式，而不是替誰加冕。

這正是「HTML 簡報」的立論基礎

「HTML 簡報」專案落在 HTML 勝出 那側 —— 給人看、要視覺與排版的最終產出。
而你正在看的這份簡報本身，就是一份自包含、可互動的 HTML —— 以身作則。
