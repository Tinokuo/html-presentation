核心主張
「我已經幾乎不用 Markdown 了 —— 改用 HTML。」
超過約 100 行的 Markdown，他坦言「幾乎讀不下去」。
這不只是個人偏好 —— Anthropic 內部已把 HTML 當成 plan、code review、設計系統、報告的預設格式。
想法很簡單：讓 Claude 直接產出能排版、視覺化、可互動的東西，而不是一堆純文字符號。
他提出的五大理由

為什麼是 HTML？

01 資訊密度

表格、CSS、SVG 圖、程式碼、互動元件、圖片——通通塞進一份檔案。

02 長文可讀性

Markdown 過 100 行就難讀；HTML 有結構、分頁、插圖與響應式排版。

03 易於分享

一個連結就能看；Markdown 通常還得當附件寄。

04 雙向互動

滑桿、旋鈕、可編輯欄位、複製按鈕——讀者能動態探索內容。

05 愉悅感

讀一份精緻 HTML 的體驗，和盯著一堆 Markdown 符號完全不同。

爭議與反方

代價：token，以及「給機器讀」的場景

Token 成本：同內容 HTML 約是 Markdown 的 2–3 倍（3000 字文章：~2,800 → ~8,000 tokens）。
機器理解 / RAG：要模型「讀進去」時 Markdown 反而較好（表格抽取 60.7% vs 53.6%；RAG 準確率最多 +35%）。
社群出現反諷文〈The Unreasonable Ineffectiveness of HTML〉等反駁。
作者本人承認 token 代價，但認為「表達力更強 + 讀者真的會去讀」值得——尤其 Opus 有 100 萬 token 的 context。
社群共識（2025–2026）

不是二選一，是看用途分工

／／ Markdown 勝出
當作模型「輸入」、通用指令
  • 省 token、成本低
  • 可 diff、版本好管理
  • 到處都能正常顯示
  • RAG / 結構化抽取較準
</> HTML 勝出
給「人」看的最終產出
  • 報告、簡報、結構豐富的文件
  • 版面、色彩、視覺化
  • 互動元件與分享連結
  • 長文的可讀性

重點是「配合產物的生命週期」選格式，而不是替誰加冕。

與本專案的關聯

這正好是「HTML 簡報」專案的立論基礎

「HTML 簡報」專案落在「HTML 勝出」那一側——給人看、要視覺與排版的最終產出。
Thariq 那篇，某種程度上就是這類做法的「官方背書」。
而你正在看的這份簡報本身，就是一份自包含、可互動的 HTML——以身作則。
